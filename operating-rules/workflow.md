# Workflow

## Purpose

This document defines how work moves through the Sovereign Asset System.

Its purpose is not merely procedural.

Its purpose is to preserve strategic coherence, structural discipline, GitHub-centered control, and long-horizon asset quality as the repository grows.

A strong repository does not depend on inspiration alone.

It depends on workflow clarity.

Without workflow discipline:

- good ideas fragment
- repository structure drifts
- publishing logic becomes confused
- assets are worked on inconsistently
- communication posture weakens
- value-building loses continuity

This workflow exists to prevent that.

## Core Workflow Principle

The repository must behave like a governed system, not a loose collection of documents.

Work should move through clear layers:

1. doctrine
2. context
3. classification
4. assessment
5. narrative
6. buyer-fit
7. pitch logic
8. engagement posture
9. negotiation posture
10. build direction
11. controlled commercialization if justified

No serious asset should jump randomly across these layers without reason.

## GitHub-First Operating Rule

GitHub is the sole operational source of truth.

This is a foundational workflow rule.

All strategic changes, content evolution, structural decisions, templates, skills, and operating logic must originate from the repository and remain traceable there.

### This means:

- local work may begin anywhere appropriate
- canonical repository state lives in GitHub
- version history must remain clear
- commit history must carry operational meaning
- repository structure must remain stable
- publishing logic must remain explicit

### This also means:

- do not make external systems the hidden operational core
- do not rely on silent parallel workflows
- do not let production logic drift away from repository logic
- do not normalize undocumented off-repo decisions

GitHub is not just a storage layer.

It is the governing operational spine.

## Separation of Operational Layers

Three layers must remain conceptually separate at all times:

### 1. Source Layer
The files, templates, contexts, rules, and working materials that define the system.

### 2. Build Layer
The transformation layer that generates output or structured deliverables from source.

### 3. Deployment Layer
The publishing or delivery surface that exposes output to the outside world.

These layers may connect, but they must not be confused.

### Never blur the following:

- source structure with published structure
- repository layout with deployment layout
- internal strategic files with external buyer-facing materials

A repository becomes fragile when these distinctions collapse.

## Canonical Output Root Rule

Every asset that is actually built must have one declared canonical output root.

This rule exists to prevent structural drift.

### Approved principle

For any built asset, define clearly:

- where source lives
- where generated output lives
- which directory is the canonical deployable output
- which system publishes that output

### Workflow requirement

Do not allow accidental secondary output trees to emerge.

Do not silently split production logic across:

- root
- docs
- public
- dist
- build
- output

unless that split is explicitly governed and documented.

### Correct standard

One asset, one declared output logic, one declared canonical publish path.

If that path changes, document the change deliberately.

Do not let it happen by drift.

## External Infrastructure Rule

External infrastructure may support the asset, but it must not silently replace the operational core.

### Approved hierarchy

1. local generation or structured repository work
2. GitHub commit and version control
3. GitHub-native automation or publishing logic
4. external DNS, security, or edge layer if justified

### Practical implication

Cloudflare, or any equivalent external system, should typically be treated as:

- DNS layer
- security layer
- TLS layer
- edge-control layer

It should not become the hidden publishing core unless there is explicit reason and explicit documentation.

### Reason

The more external operational dependency is placed inside the publishing core, the more fragile automation becomes.

The preferred workflow is repository-centered, traceable, and internally governable.

## Repository Evolution Order

The repository should evolve in the following order.

Do not invert this sequence casually.

### Phase 1: Doctrine and Context
Establish:

- doctrine
- portfolio context
- taxonomy
- priority framework

### Phase 2: Skills
Establish the core strategic skills that govern asset interpretation and action.

### Phase 3: Templates
Establish standardized operating documents.

### Phase 4: Operating Rules
Formalize repository governance.

### Phase 5: Asset-Level Application
Apply the system to real domains.

### Phase 6: Expansion
Add more specialized layers only after the core is stable.

This sequence prevents premature scaling.

## Asset Workflow

Every serious asset should move through a controlled lifecycle.

### Stage 1: Entry
The asset is recognized as worth repository attention.

Typical actions:

- create or plan a dossier
- assign initial portfolio role
- assign provisional maturity stage
- determine whether deeper work is justified

### Stage 2: Classification
The asset is classified through the portfolio taxonomy.

Typical actions:

- assign sector class
- assign geographic orientation
- assign asset mode
- assign buyer type
- assign commercialization orientation
- assign priority tier

### Stage 3: Strategic Assessment
The asset is assessed through `domain-strategic-assessment`.

Typical outputs:

- strategic verdict
- development depth
- revenue tolerance
- portfolio role
- immediate next action

### Stage 4: Narrative Design
If justified, the asset proceeds to `strategic-narrative-design`.

Typical outputs:

- conceptual center
- authority claim
- narrative posture
- development translation
- narrative thesis

### Stage 5: Buyer Fit
If justified, the asset proceeds to `buyer-fit-analysis`.

Typical outputs:

- primary buyer class
- ownership logic
- buyer timing
- outreach posture implication
- buyer-fit verdict

### Stage 6: Pitch Logic
If justified, the asset proceeds to `executive-pitchwriting`.

Typical outputs:

- buyer-facing objective
- ownership thesis
- pitch form
- tone and compression
- pitch readiness verdict

### Stage 7: Engagement Posture
If buyer-facing motion is justified, the asset proceeds to `buyer-engagement-posture`.

Typical outputs:

- first-contact posture
- continuation posture
- silence handling logic
- engagement verdict

### Stage 8: Negotiation Posture
If real buyer seriousness exists, the asset proceeds to `premium-asset-negotiation`.

Typical outputs:

- price posture
- flexibility boundary
- response strategy
- walk-away logic
- negotiation verdict

### Stage 9: Build Direction
If development is the correct path, the asset moves toward build planning and implementation.

### Stage 10: Ongoing Review
The asset may be reclassified, strengthened, held, or re-entered into buyer-facing logic later.

## Value Construction Workflow Rule

The asset workflow must not be interpreted as a passive filter for deciding whether an asset is worthy.

Its purpose is also to identify how an asset can be strengthened.

This means:

- portfolio entry should identify the best path for value development
- strategic assessment should identify the highest-value trajectory
- narrative design should deepen strategic legitimacy
- buyer-fit analysis should improve ownership logic, not merely test it
- dossier creation should consolidate the layers that increase authority and asset maturity

The workflow is therefore not only classificatory.

It is constructive.

The repository should help move selected assets from latent value to clearer strategic value.

## Workflow by Asset Condition

Not every asset should move through the full sequence.

### Dormant Hold Workflow
Use when the asset is valuable but not ready for active investment.

Typical path:

- classify
- assess
- hold
- review later

### Positioning Workflow
Use when the asset deserves strategic framing but not deep build.

Typical path:

- classify
- assess
- narrative design
- light dossier
- hold or selective outreach later

### Sale Candidate Workflow
Use when buyer logic is relatively strong.

Typical path:

- classify
- assess
- buyer fit
- pitch logic
- engagement posture
- negotiation if warranted

### Build Candidate Workflow
Use when the asset deserves deep development.

Typical path:

- classify
- assess
- narrative design
- build direction
- authority-layer development
- selective commercialization or outreach later if justified

### Yield Support Workflow
Use only when recurring income can coexist with dignity.

Typical path:

- assess
- narrative check
- revenue tolerance check
- commercialization logic
- controlled build or monetization layer

Do not force yield logic onto every asset.

## Dossier Workflow

If an asset becomes active enough to justify a unified operating file, create its `domain-dossier.md`.

The dossier becomes the canonical internal working document for that asset.

### Dossier rule

A dossier should consolidate:

- classification
- assessment
- narrative
- buyer-fit
- pitch direction
- engagement posture
- negotiation posture
- build direction
- current decision

Do not scatter core asset logic across random notes once the dossier exists.

The dossier is the operating center for that asset.

## Buyer-Facing Workflow

External communication must follow a stricter logic than internal analysis.

### Approved order

1. confirm asset readiness
2. confirm buyer fit
3. confirm pitch readiness
4. confirm engagement posture
5. send one strong first contact if justified
6. continue only on signal
7. deepen only when seriousness is earned
8. negotiate only when strategic conditions justify it

### Disallowed order

- contact first, think later
- follow up repeatedly to create motion
- negotiate before buyer logic is solid
- reveal price before ownership logic is legible
- use pressure to compensate for weak positioning

## Silence Handling Workflow

Silence is not a workflow error.

If first contact receives no meaningful response, the default workflow is not repeated pursuit.

### Default posture

- log the contact
- accept silence
- do not automatically follow up
- continue strengthening the asset
- revisit only after meaningful strategic change if justified

### A justified later re-entry may require:

- stronger asset development
- better buyer clarity
- changed market timing
- a materially improved buyer-facing brief
- a meaningful change in the ownership case

Repeated contact without changed conditions is not disciplined workflow.

## Build Workflow

When an asset moves into build mode, the following sequence should govern:

1. confirm build depth
2. confirm narrative translation
3. confirm intended output logic
4. confirm canonical output root
5. confirm deployment logic
6. build in a way consistent with doctrine
7. review before publishing
8. publish through the declared GitHub-centered flow
9. record asset maturity change if warranted

### Build discipline

Do not build merely because an idea is attractive.

Build because the asset justifies structure.

## Publication and Monetization Gates

Before any asset output is published, and before any recurring income logic is activated, the asset must pass explicit decision gates.

These gates exist to prevent conceptual drift, premature commercialization, and damage to long-horizon value posture.

### Gate 1: Structural Stability

Ask:

- Is the asset's formal positioning clear?
- Is the architecture coherent enough for public exposure?
- Does the build reflect the doctrine rather than contradict it?
- Is the asset stable enough to be seen as intentional rather than provisional?

If not, do not publish yet.

### Gate 2: Conceptual Integrity

Ask:

- Does the public-facing expression remain faithful to the asset's highest-value reading?
- Has the narrative become clearer, or merely louder?
- Is the asset still being framed at the right level of seriousness?
- Has any part of the build cheapened the conceptual structure?

If conceptual integrity is weak, revise before publication.

### Gate 3: Authority Readiness

Ask:

- Does the asset look credible to a serious outside observer?
- Does the content support authority rather than imitation?
- Are the definitions, claims, and framing defensible?
- Does the asset appear governed rather than improvised?

If authority readiness is weak, strengthen before publication.

### Gate 4: Revenue Compatibility

Ask:

- Would recurring income strengthen the asset, or redefine it downward?
- Is the proposed revenue model aligned with the asset's dignity?
- Would this revenue path damage future acquisition quality?
- Is revenue remaining secondary to value maximization?

If the revenue model introduces degradation, do not activate it.

### Gate 5: Anti-Mobility Check

Ask:

- Is this move being made because it is strategically correct?
- Or because speed, impatience, or activity pressure is driving it?
- Is the asset actually ready, or merely available?

If the move is driven by impatience rather than correctness, stop.

## Gate Outcomes

Each gate should produce one of the following outcomes:

- Pass
- Revise before proceeding
- Delay until stronger maturity
- Reject this path

Publication should not happen because material exists.

Monetization should not happen because revenue is possible.

Both should happen only when the asset remains stronger after the move than before it.

## Final Rule

No publication or monetization step should be taken if it weakens:

- strategic clarity
- authority potential
- dignity
- long-horizon sale quality
- conceptual coherence

A sovereign-grade asset must become more legible and more defensible when exposed.

If exposure weakens it, exposure was premature.

- template revision

- operating rule revision

- asset dossier revision

This improves repository legibility and commit discipline.

## Commit Discipline

Commits should reflect real operating meaning.

### Good commit characteristics

- specific
- traceable
- limited in scope
- readable later
- aligned with the repository structure

### Preferred examples

- `docs: add portfolio taxonomy`
- `feat: add domain strategic assessment skill`
- `docs: add buyer brief template`
- `docs: define workflow rules`
- `refactor: strengthen naming conventions`
- `docs: create domain dossier template`

### Avoid commits such as

- `update stuff`
- `misc changes`
- `new file`
- `fix things`
- `more edits`

A weak commit history weakens the operating system.

## Review Gates

Before considering any major addition stable, pass it through these checks.

### Check 1: Doctrine Alignment
Does this align with value maximization, dignity, and sovereign-grade standards?

### Check 2: Structural Clarity
Does this improve structure, or create ambiguity?

### Check 3: Cross-Asset Reusability
Will this remain useful across many assets, not just one moment?

### Check 4: Workflow Fit
Does it belong in the current stage of repository evolution?

### Check 5: Quality Standard
Would this still look credible six months from now?

If the answer is weak, revise before committing.

## Anti-Drift Rules

The repository must actively resist drift.

### Drift type 1: Conceptual Drift
Occurs when assets begin being treated like ordinary inventory.

### Drift type 2: Structural Drift
Occurs when files, naming, layers, or output paths become inconsistent.

### Drift type 3: Workflow Drift
Occurs when steps begin being skipped, reversed, or improvised without reason.

### Drift type 4: Communication Drift
Occurs when buyer-facing behavior becomes more aggressive, cheaper, or more reactive than the doctrine allows.

### Drift type 5: Commercial Drift
Occurs when secondary income or early sale pressure starts redefining the system.

Every meaningful workflow decision should reduce drift, not increase it.

## Reassessment Workflow

Reassessment is legitimate when:

- the market changes
- the asset matures
- the narrative improves
- buyer clarity improves
- build quality changes the value case
- revenue logic changes
- outreach outcome reveals new information

Reassessment is not a sign of instability.

It is part of serious portfolio governance.

But reassessment must be documented, not improvised silently.

## Documentation Rule

If a strategic decision affects how an asset should be classified, built, sold, held, or communicated, document it.

Do not rely on memory.

Do not rely on scattered chat fragments.

The repository must become the memory system.

## Final Workflow Rule

When uncertainty appears, do not compensate with speed.

Compensate with clarity.

If the asset is not ready, do not force motion.

If the buyer is not serious, do not force engagement.

If the structure is not clean, do not force scale.

The correct workflow is not the one that creates the most movement.

It is the one that protects strategic value while making deliberate progress.
